1 # Move from root to FairMQ
2 
3 This example uses standard FairRoot analysis chain:
4 
5 * detector simulation,
6 * digitization,
7 * reconstruction (hit finding, tracking, track fitting),
8 
9 to show how to switch for root single-core processing to FairMQ multi-core pipeline processing.
10 
11 ## MQ Devices
12 
13 The following MQ devices are implemented:
14 
15 - FairMQPixelSampler - reads data branch names specified by AddInputBranchName(string) from the input file(s) set by AddInputFileName(string) via FairSource. Currently one may attach FairFileSource (chains of ROOT files), PixelDigiSource (ascii files with PixelDigis) or PixelDigiBinSource (binary files with PixelDigis);
16 
17 - template <typename T> FairMQPixelTaskProcessor - runs a task of class T. The class T needs to have the following functions:
18 void GetParList (TList* parList); - used to get the list of parameter objects needed by class T;
19 void InitMQ (TList* parList); - initialization of the class T object using the parameters;
20 void ExecMQ (TList* inputList, TList* outputList); - analyze data from inputList and store the result in outputList;
21 
22 - FairMQPixelFileSink creates output ROOT file set by SetOutputFileName(string) and tree, with the branches specified by AddOutputBranch(string className, string branchName). Currently only two classes (FairEventHeader and TClonesArray of any objects) are allowed:
23 fileSink.AddOutputBranch(“FairEventHeader”,”EventHeader.”);
24 fileSink.AddOutputBranch(“TClonesArray(anyclassname)”,”branchname”);
25 
26 The above MQ devices sends FairMQParts multi-messages with TMessages inside. This method is not very efficient, but is straighforward and generic to use.
27 
28 Besides, there are also binary-transport versions of the above devices:
29 
30 - FairMQPixelSamplerBin
31 - FairMQPixelTaskProcessorBin
32 - FairMQPixelFileSinkBin
33 
34 but only some data classes may be sent (the ones specified in PixelPayload).
35 
36 FairMQPixelMerger - device to merge different parts of events. The event parts are merged into one object and sent to output.

38 ## Executables
39 
40 Separate executables for the different devices have been created:
41 
42 - **pixel-sampler** ( **ex-samplerBin**),
43 - **pixel-processor** ( **ex-processorBin**),
44 - **pixel-sink** ( **ex-sinkBin**),
45 
46 which run given devices and allow setting of the different parameters from the command line. These devices may run in different control modes:
47 - interactive - waiting for the keyboard input to change states;
48 - static - changes states automatically;
49 - dds - react to the commands sent via dds_intercom.

51 ## Topologies and scripts
52 Several shell scripts using different topologies are implemented:
53 
54 #### ./startFairMQPixel.sh
55 - **topology:** sampler-3processors-sink
56 - **functions:** can process different task with (--task-name PixelFindHits, PixelFindTracks, PixelFitTracks),
57 can set the number of processors tasks with -p, can limit number of events with -m,
58 can change running mode with --command (interactive or static)
59 - **preparations:** root -l -q 'run_sim.C(100000)' &> sim_100k.dat; root -l -q run_digi.C;
60 
61 #### ./startFairMQPixel_3Levels.sh
62 - **topology:** sampler-2proc-2proc-2proc-sink
63 - **functions:** runs hit finding, track finding, track fitting
64 can limit number of events with --m
65 - **preparations:** root -l -q 'run_sim.C(100000)' &> sim_100k.dat; root -l -q run_digi.C;
66 
67 #### ./startFairMQPixelBin.sh
68 - **topology:** sampler-5processors-sink
69 - **functions:** run PixelFindHits
70 transport binary data between devices
71 - **preparations:** root -l -q 'run_sim.C(100000)' &> sim_100k.dat; root -l -q run_digi.C;
72 
73 #### ./startFairMQPixelBinProxy.sh
74 - **topology:** 2samplers-proxy-5processors-proxy-2sinks
75 - **functions:** run PixelFindHits
76 transport binary data between devices
77 proxy has several input and output queues
78 - **preparations:** root -l -q 'run_sim.C(100000)' &> sim_100k.dat; root -l -q run_digi.C;
79 
80 #### ./startFairMQPixelMerger.sh
81 - **topology:** 3samplers-3processors-merger-sink
82 - **functions:** data is read by 3 samplers from 3 input files, for each station separately
83 hit finding is run on processors separately for each station data
84 merges event data from different stations and sends whole events to the sink
85 - **preparations:** root -l -q 'run_sim.C(100000)' &> sim_100k.dat; root -l -q run_digi.C; root -l -q run_digiToBin.C;
86 
87 #### ./startFairMQPixelSimulation.sh
88 - **topology:** 3samplers-sink
89 - **functions:** samplers are producing Monte Carlo data, one can set
90 the number of events produced by each sampler using --nof-events, the default
91 TGeant3 can be changed to --transport-name TGeant4

93 ## Running with DDS
94 To run the example in the DDS, please check http://dds.gsi.de for the installation instructions.
95 The next step, you have to configure fairroot with -DDDS_PATH="/path/to/dds/install/dir/" and rebuild it.
96 
97 These programs are started with the pixel-dds-topology.xml topology, that can be started in the following way:
98 
99 ```bash
100 dds-server start -s
101 dds-submit --rms localhost -n 10
102 dds-topology --activate bin/examples/MQ/pixelDetector/pixel-dds-topology.xml
103 ```
104 
105 One can check the status or control the devices using:
106 
107 ```bash
108 ./bin/fairmq-dds-command-ui
109 ```
110 
111 There is also a interactive shell script created to ease the devices' state control. Different states are marked with different colors, and by pressing appropriate keys it is possible to control the devices:
112 
113 ```bash
114 user@host:build$ ./bin/examples/MQ/pixelDetector/controlDDS.sh
115 RUNNING->s->READY->t->DEVICE_READY->d->IDLE->q /x-Exit
116 [parmq-server][processor_0][processor_1][processor_2][ sampler][ sink]
117 ```
118 
119 Once the sampler finishes stops running, the DDS server may be stopped:
120 
121 ```bash
122 dds-server stop
123 ```
