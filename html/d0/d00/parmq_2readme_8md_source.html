1 ParameterMQServer
2 ===============
3 
4 The ParameterMQServer device sends out parameter objects (serialized with TMessage) for incoming requests (parameter name & run ID). The communication is via REQ-REP pattern.
5 
6 The device executable has to be started with the following command line parameters:
7 
8 - `--id <device id>` device id (has to match one in the json file).
9 - `--mq-config <path/to/device-config>` location of the transport config file.
10 - `--first-input-name` location of the first input file for the FairRuntimeDb.
11 
12 Optional options are:
13 
14 - `--first-input-type arg (="ROOT")` first input file type (ROOT/ASCII).
15 - `--second-input-name arg (="")` location of the second input file.
16 - `--second-input-type arg (="ROOT")` second input file type (ROOT/ASCII).
17 - `--output-name arg (="")` location of the output file.
18 - `--output-type arg (="ROOT")` output file type (ROOT).
19 - `--channel-name arg (="data")` output channel name.
20 
21 When both inputs are provided, and a parameter exists in both, first input will be taken.
22 
23 For further documentation of parameter system check [parameter section in FairRoot webpage](https://fairroot.gsi.de/?q=node/28).
24 
25 The request for parameters is a string in this form: `"ParameterName,RunID"`.
26 
27 For an example client device that retrieves the parameters from the ParameterMQServer, take a look at [../examples/MQ/7-parameters/](`../examples/MQ/7-parameters/`).
